# 前端开发规范

## 适用范围

说明前端界面、组件、交互、样式，以及 OpenPrd 内置前端设计资产框架的项目级约定。

## 设计前置顺序

界面任务不要直接从写代码开始，默认先完成下面几步：

1. 选 `lens`
2. 写清用途、受众、气质端点、约束和记忆点
3. 选 `theme`
4. 选 `layout skeleton`
5. 选高频 `component set`
6. 判断是否需要 `facts-sheet`
7. 判断是否需要 `asset-spec`
8. 判断是否需要 `image-preflight`
9. 没参考方向时先做 3 个异源方向
10. 用户选定方向后再进入实现

这些资产默认放在 `.openprd/design/`。

## 审美框架

- `lenses/`：定义本次页面的设计判断角度，例如编辑感、发布感、工具感、沉浸展示感。
- `themes/`：冻结颜色、文字、圆角、边框、阴影和密度，不在实现中途临时发明。
- `layouts/`：提供页面骨架，避免每次从空白页拼版。
- `components/`：提供高频结构块，减少一页一个新写法。
- `recipes/`：按任务类型给默认组合。
- `checklists/`：提供 P0/P1/P2 自检门。
- `anti-slop.md`：反模板化检查，避免默认紫白渐变、通用字体、白底卡片阵列和无语境装饰。
- `templates/`：提供空白工作区可直接起步的页面模板，适合静态原型、首页和首屏实验。

## 前端体验规则

- 大界面任务默认要有 3 个明显不同的候选方向，不能只是同一种安全解的轻微变体。
- 每个方向都要有具体审美主张和用户记忆点，不要只写“现代、简洁、高级”。
- 没有品牌、参考图或设计系统约束时，不要默认使用紫白渐变、蓝紫光晕、玻璃卡、Inter/Roboto/Arial/system-ui 通用字体栈或白底卡片堆叠。
- 字体、色彩、动效、表面/背景和构图要服务同一个气质；复杂页面可以更有层次，克制页面则必须在比例、排版和状态细节上精确。
- 工具型页面优先扫描效率、层级和稳定密度，不做营销站式夸张排版。
- 内容型页面若真实图片是成立前提，必须先解决图片问题，再做大版式。
- 产品页、发布页、内容页、deck、招商页等若写到具体事实，先核实再实现。
- 一旦方向已选，先锁定 `lens + theme + layout + component set + 审美主张 + 记忆点`，再进入编码。

## 图片与效果图

- 纯图片、封面、上架图、海报、插画、图标、贴纸、主视觉、mockup 或先看样子，默认走 Codex 原生 Image 2。
- Image 2 是工具路径，不是审美豁免；生成前必须写清用途、受众、气质、约束和记忆点。
- prompt 里要带上审美主张和 anti-slop 约束，避免默认紫白/蓝紫渐变、通用字体、白底卡片堆叠和无语境装饰。
- 生成图先是候选效果图；只有用户确认后，才进入后续 reference-set 或实现对比。
- 看候选图时不只判断是否生成成功，还要判断气质、层级、字体/色彩/表面角色和记忆点是否成立。

## 界面结构

- 书页式阅读器优先使用“目录 + 当前页正文”的结构，目录只保留章节和章节内结构节点，不把具体练习题当成目录叶子。
- 当前页正文可以在自身容器内滚动，但页眉、标题、控制区应保持较轻，避免和正文争夺视觉重心。

## 交互规范

- 章节切换优先采用换页语义，而不是让整篇内容形成长卷滚动。
- 阅读进度、上一章、下一章、字号控制应服务于正文阅读，不应把控制区做成主视觉。

## 验收规则

- 已有参考图时，用 `openprd visual-prepare` 和 `openprd visual-compare` 留下视觉证据。
- 无参考图的大界面任务先做方向评审，再实现。
- 视觉证据不仅检查结构是否摆对，也要检查选定气质、字体/色彩/动效/表面层级和记忆点是否保住。
- 轻量 UI 可视优化也要视觉证据：卡片宽度、间距、留白、对齐、颜色、圆角、字号、按钮和图标等小改，动手前要有一句审美意图和记忆点，收口时用修改前后对比图、局部焦点证据板、截图实测证据板或对齐辅助线证据板复核。
- 新功能或改动包含同构列表、卡片、网格或表格时，即使用户没有主动反馈“没对齐”，也默认同时检查容器轨道和内部内容槽位：容器轨道包括卡片外框、列宽、行顶、间距，内部槽位包括标题、副标题、标签、描述、状态、价格、按钮、图标、操作区等相同文案类型/相同组件槽位的 x/y/宽高/baseline spread，并用 `openprd visual-compare . --board <alignment-board.json>` 留下辅助线证据。只量外框、列宽或行顶不算完整对齐验收。
- 单个 logo、icon、avatar、badge、按钮图形或图片内部需要居中判定、视觉重心评估或偏心排查时，默认裁出目标元素并用 `openprd visual-compare . --board <centering-board.json>` 留下内部居中证据。证据板要显示红色画布中心线、绿色主体外接框、黄色视觉重心点，并记录主体外接框中心偏移和视觉重心偏移；单张截图或“看起来居中”的主观判断不能替代。
- build、package、`openprd dev-check`、单元测试和单张原始截图不能替代视觉证据。
- 空白工作区的静态原型默认先从 `.openprd/design/templates/` 里挑最近模板，再按选中的 `lens + theme + layout + component set` 改写。

## 维护规则

- 每次新增界面模式、组件规范、设计骨架或交互规则后，必须同步更新本文件和 `.openprd/design/`。
