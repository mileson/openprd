# 产品流程说明

## 核心流程

OpenPrd 的核心流程是：用户在项目中初始化或接入 `.openprd/` 工作区，Agent 先读取 `openprd run --context`、`status`、`next` 等只读状态，再用 `openprd-requirement-intake` 按影响面、未知数、决策成本和验证成本判断当前输入属于直接处理、现有功能优化还是新功能/新流程方案，并在内部保留 L0/L1/L2 路由码。当前流程强调五层协作面：最前面新增独立脑暴模式，当用户明确说“帮我梳理下”“先想清楚”“进入脑暴模式”，或一句话需求明显还需要先收敛业务方向、用户、商业目标、竞品和内部复用时，先进入 `openprd brainstorm`，汇总 benchmark、knowledge、当前工作区和开放问题，生成稳定 `brainstorm.html` 工作台。脑暴模式不再只做需求整理，而是默认追问和呈现“现在怎么解决、为什么现在要换、谁拍板/谁受影响、第一批最容易触达的社区/种子用户、先不做完整产品时怎么手工交付、手工作战卡、一件事 MVP、周末级验证、如果必须开始做产品也只自动化最重复的一步并先压成 forms / lists / CRUD 骨架、第一批客户路径、从第一个客户开始怎么收费、客户 1 如何打平成本、什么承诺才算真需求、有没有 10 个样本和更强付费信号、最低成本验证、先活下来约束、达到什么条件才允许产品化、增长纪律、可逆性、客户真问题、价值观一致性、止损线和可复用基础”；确认方向后，再进入需求入口分流。分流之后，L2 继续通过 `intake-reflection.md` 做需求入口自省，并把 `clarify` 压缩成对话内轻量确认或简短清单；只有 requirement 摘要里的 target、scope、out-of-scope 和 acceptance 已被用户确认，或用户明确表示“不需要进行任何确认”时，才允许继续 `capture/classify/synthesize` 写入路径，再通过 `review.html` 帮用户看清楚、比较清楚、确认清楚。对于仍处在 0 到 1 探索、值不值得做或创业验证阶段的 L2，requirement 摘要除了“需求判断 / 需求理解 / 功能范围 / 技术方案”，还要主动补一层“验证与创业闭环”：当前替代方案、第一批先找谁、先怎么手工交付、手工作战卡、第一版只做哪一件事、周末级验证、第一批客户路径、从第一个客户开始怎么收费、客户 1 如何打平成本、什么承诺才算真需求、有没有 10 个样本和更强付费信号、最低成本先验证什么、能不能先用 spreadsheet / 表单 / no-code 跑起来、如果必须开始做产品也只自动化最重复的一步并先压成 forms / lists / CRUD 骨架、达到什么条件才允许产品化、增长阶段守什么纪律、验证阶段怎样先活下来，以及这条路是否可逆、是否真在解决客户问题、是否符合团队价值观，以及这是不是团队愿意长期住进去、不会反过来绑住自己的业务形态。界面、页面、视觉、样式和前端体验任务在真正进入实现前，还要额外经过一层设计前置门：读取 `$openprd-frontend-design` 和 `.openprd/design/`，先补 `facts-sheet`、`asset-spec`、`image-preflight`、`direction-plan`、`selected-direction` 这组活动合同，避免事实瞎编、素材空心和 3 个方向都收敛成同一种安全极简解。若用户已经提供效果图、设计稿、参考截图或其他明确参考图，默认把它当成主参考源；现有 starter、样式库和 design tokens 只在足够接近时作为加速器，不得反过来主导最终风格。空白前端冷启动如果用户原话已经明确页面主题、模块范围或“直接实现”的意图，优先用 `openprd run --context --message <用户原话>` 对齐路径；`design-starter` 生成入口文件后，默认立即进入 `Patch Mode`，围绕同一路径继续细化页面，而不是 delete-first 后再重建。`Patch Mode` 的完成定义也要收紧：不只是在对话里说明计划、下载素材或补合同，而是要把最终结构、样式、真实图片或参考约束真正落进入口文件，清掉主要占位，再同步 active design artifacts。执行段通过 tasks、loop、回归报告和必要的视觉评审证据把实现与验证沉淀成结构化证据，并用 `openprd-test-strategy` 为任务选择单元、集成、端到端、人工、视觉、小程序、性能或安全验证组合，同时用 execution strategy 为任务标注 `serial / parallel-workers / parallel-workers-isolated`、`write-scope`、`owner-role`、`local-verify` 和 `integration-owner`；质量段通过 `openprd quality --verify` 生成 HTML 回归测试报告，优先展示整体回归结果、逐需求模块结果、测试块通过情况、分层测试策略矩阵、未通过项和需确认遗漏。用户只是要求生成图片、封面图、配图、海报、插画、图标、贴纸、头像、banner、主视觉/KV、运营图、确认用效果图、视觉稿、mockup 或“先看样子”时，Agent 应直接调用 `imagegen`，也就是 Codex 原生 Image 2，生成图片；只有实际发生 `imagegen` 调用后，才能汇报生图结果、失败或限流。大界面改动在需求分流后、PRD 定稿或实现开工前，先用 Codex Computer Use 截取产品内当前功能截图，再用 `imagegen` 生成至少 3 个设计方向，并拼接成带 1/2/3 序号的横向评审大图给用户选择；这 3 个方向还要在 `.openprd/design/active/direction-plan.md` 中明确写出不同的 lens、theme、layout 或素材策略，不能只是小幅换色或轻微留白差异。如果确认参考图是一张整板、网格图或多对象组合，再先走 `visual-prepare` 生成 reference-set、contact sheet 和 compare-plan，然后才进入逐项实现与验收；`review.html` 不替代图片或效果图生成，`visual-compare` 只在进入实现阶段后用于视觉证据：已有参考图时对比效果图和实现截图，无参考图但改动界面时对比修改前和修改后截图。EVO 只作为内部质量评估/验证层代号，不作为用户读报告的前置概念。执行阶段的单个任务只保留最小足够 task evidence；阶段收口或整体交付前再经过 standards 和 quality 门禁，确认基础文档、文件说明书、文件夹说明书、日志、业务护栏、测试策略、执行模式、视觉方案确认、视觉复刻或修改前后自检证据和评估证据是否需要补齐或更新。

## 用户路径

- 初始化项目：用户运行 `openprd init` 或 `openprd setup`，生成 `.openprd/`、`docs/basic/`、AGENTS 和各端 agent guidance；`.openprd/README.md` 与模板层 `README.md` 默认使用简体中文，英文切换入口写到 `README_EN.md`，如需兼容旧外链可保留 `README_CN.md` 作为跳转页。仓库首页 README、对外概览和带文字的示意图，也默认先按用户可见需求类型解释“直接处理 / 现有功能优化 / 新功能/新流程方案”，优先说明用户场景、确认方式、收益和风险；如果提供英文版，文字图片也要成对维护。Codex hook 默认使用 `lite`，在明确 OpenPrd/深度流程提示词和新产品、模块、流程类需求下注入上下文，用只匹配直接编辑工具的轻量写入门禁阻断未确认需求的实现改动，并在本轮结束前通过 `Stop` 回顾是否值得沉淀项目经验。
- 规划或分析：用户要求“看看、梳理、怎么改、预计动哪些文件”时，Agent 只读检查代码、文档和 OpenPrd 状态后输出证据化结论，不启动 loop 或推进任务。
- 独立脑暴：用户明确要求先梳理、进入脑暴模式，或 Agent 判断这句话需求还需要先收敛方向时，先运行 `openprd brainstorm . --topic <当前主题> --open`。脑暴页默认把“核心诉求 / 目标结果 / 当前替代方案 / 推荐方向”抬到顶部，并补出“现状与机会 / 方向对比 / 假设与验证 / 现有基础与复用”四块主体内容；底部固定提供“继续一起梳理 / 先整理验证计划 / 按这个方向整理成 PRD”三类复制指令。如需更稳定的展示文案，再用 `openprd brainstorm-presentation . --template` 生成模板并校验写回。
- 生成图片内容：用户要求生成图片、封面图、配图、海报、插画、图标、贴纸、头像、banner、主视觉/KV、运营图、效果图、视觉稿、mockup、先看样子或确认设计方向时，Codex hook 只注入提醒，不打开需求入口写入门禁；Agent 默认调用 `imagegen`（Codex 原生 Image 2）产出图片，除非用户明确指定 HTML、SVG、CSS、Canvas、代码稿或可编辑矢量产物。未发生 `imagegen` 调用前，不得声称“生图限流”或“生图失败”。
- 大界面改动方向确认：当界面、页面、视觉、样式或前端体验需求会明显改变信息架构、核心布局、主视觉、关键路径、组件层级/密度，或用户需要先选择设计方向时，Agent 在 PRD 定稿或实现开工前先用 Codex Computer Use 进入产品内对应功能并截图，再用 `imagegen`（Codex 原生 Image 2）基于截图产出至少 3 种不同设计思想方向，横向拼接为一张左上角标注 1/2/3 的大图并展示给用户确认。用户确认方向前，不进入大 UI 实现；用户确认后，如果参考图是一张整板、网格图或多对象组合，再用 `openprd visual-prepare` 生成 reference-set、contact sheet 和 compare-plan，决定哪些对象真正进入后续实现与验收。
- 前端设计框架预检：当任务已经明确会落到真实界面实现，而不是只出候选效果图时，Agent 还要先读取 `.openprd/design/`。页面涉及产品名、版本、发布日期、规格、价格、地点或引用数据时，先补 `facts-sheet.md`；页面依赖 logo、产品图、UI 图、摄影图、插图、图表或品牌 token 时，先补 `asset-spec.md`；旅游、展览、案例、品牌故事、内容站、发布页等内容型页面，需要先判断真实图片是不是页面成立前提，必要时补 `image-preflight.md`；无明确参考方向时先补 `direction-plan.md`，用户拍板后再把选中的 lens、theme、layout、组件和主要风险写回 `selected-direction.md`。
- 澄清与确认：当需求仍模糊、只有一句话、或需要多方案比较时，Agent 优先运行 `openprd clarify`。复杂或模糊需求会先生成三轮 `intake-reflection.md`，从用户意图、项目上下文和产品质量缺口压缩出必须确认的问题；简单明确调整只做轻量项目映射，不进入完整问题墙。即使当前没有 active requirement gate，只要工作区仍处于冷启动或缺少关键字段，`clarify` 也会稳定写出 `intake-reflection.md`，并把首轮项目画像显式带回 CLI / 对话输出；仅包含 OpenPrd 自生成脚手架的 fresh init 目录会继续按 `cold-start-greenfield` 处理，而不是误判成已有项目。`clarify` 只返回 `inline` 或 `inline-with-checklist`，Agent 先在聊天里用业务和产品语言复述：这次主要给谁、在哪个场景下解决什么问题、第一版先做什么、先不做什么、不能影响什么、会牵动哪些环节、有哪些业务风险；默认先追问 1 个最高价值问题，必要时再给 2 到 3 个方向和取舍供用户选，不生成或打开澄清 HTML。`面向个人消费者场景 / 面向企业服务场景 / 以 Agent 为主要使用场景` 的 PRD 场景视角也会前移到这一步：面向个人消费者场景优先看首次场景、用户感受和回访动力；面向企业服务场景优先看拍板方、使用方、推进方和上线阻力；以 Agent 为主要使用场景时优先看自主边界、人工兜底和失败恢复。active requirement gate 存在时，当前 lane 的澄清、capture、synthesize、review 都优先读写 `.openprd/harness/session-states/<session>.json`，不再直接复用别的会话留下的 `state/current.json`。在写回 requirement 事实前，当前 lane 需要先在对话里确认 target、scope、out-of-scope 和 acceptance；只有这些摘要已被确认，或用户明确表示“不需要进行任何确认”时，才允许继续 `capture/classify/synthesize` 写入路径。合成 PRD 后会生成一个工具无关的 work unit id，把本轮需求、目标根目录、版本、摘要指纹和稳定评审页写入 `.openprd/engagements/work-units/`。Agent 生成 spec、tasks 和对外说明时跟随用户当前主语言；无法判断时使用简体中文兜底。必要专有名词、品牌名、命令名、路径、字段名和 API 术语可以保留原文；如果只是内部措辞整理，应使用 `openprd capture . --source agent-normalized` 写回，避免把无语义变化的润色当成新的用户确认。Agent 应优先把 `.openprd/artifacts/active/<version>-review/artifact.html` 这类稳定 artifact 路径给用户，而不是依赖可能被其他对话覆盖的 `engagements/active/review.html`；页面复制出的确认命令会携带 `--version`、`--digest` 和 `--work-unit`，`review` 校验一致后才记录确认状态。默认 approval policy 是 `decision-points`：当前 lane 仍需要人类决策时，先让用户评审当前稳定 artifact，再记录确认；只有当用户明确表示“不需要进行任何确认”时，lane 才能进入 `silent-record`，跳过 requirement 摘要确认并只对当前精确匹配的 artifact 直接记录 review，单纯的“请帮我实现/继续实现”或“不要评审”都不触发这个豁免。如果用户刚刚确认的是现有功能优化（L1）的 mini-plan、范围边界或正式产品边界，后续承接文案要写成“已确认，我按这个继续”，不要写成“确认，我们就按这个……”这种像再次索取确认的句子。已合成版本后再次 `capture` 用户确认或推导得到的 PRD 内容，会把旧评审状态标记为过期，并清掉 active 版本指针，防止多对话混用旧评审 artifact；仅用于 `agent-normalized` 的内部措辞规范化或 `reviewPresentation` 展示层写回，不会重开用户 review。未记录 review 或标记为 `needs-revision` 的 PRD 不能生成 OpenPrd change 或进入实现；当前 artifact 记录完成后应先运行 `openprd change --generate` 产出 specs 和 tasks。若用户原始意图已经明确要求实现，tasks 就绪后可直接继续执行；否则先展示执行确认清单，列出本轮目标、将执行内容、不做事项、验证方式和已知风险，再请求明确执行授权。
- 业务护栏澄清：当 PRD 命中免费用户、额度、AI 生成、模型调用或第三方成本风险时，`next` 会把成本来源、额度限制、滥用防护、监控信号、报警阈值和止损动作列为待确认问题。
- 大量只读扫描：当用户明确要求深度调研、全面梳理、交叉验证、并行排查或对标复刻时，OpenPrd discovery skill 负责判断是否启动只读 subagent，并要求主 Agent 汇总证据后再写入 claim、requirements、specs 或 tasks。
- 实现或继续任务：用户明确要求开发、实现、继续、深度调研、深度对标或复刻落地时，Agent 才能运行执行命令。`openprd run --context` 会优先推荐已确认 PRD 的 change 生成；如果已有 active change，则按任务 `type` 统计实质实现任务，并给出执行模式建议：L0 或小范围修正默认保持 `serial`，由主 Agent 直接收口；中等规模的 L1/L2 任务、且 `implementation` 任务达到 3 个时，建议使用 `parallel-workers`，由主 Agent 分配写入范围清晰的 worker shard，再统一 review、integrate 和总验证；`implementation` 任务达到 10 个时再升级到 `parallel-workers-isolated`，建议使用独立 worktree 或等价隔离环境，并通过单任务 Loop 会话推进。显式继续历史会话时，`run --context` 先查 `~/.openprd/registry/sessions.jsonl`，先找回真实 workspace，再从目标工作区的 session binding、requirement gate、事件和 work unit 线索恢复该需求线；只要恢复到别的 workspace、或命中了显式 session / task-handle / work-unit，默认都升级为隔离执行建议，不再与当前 active change 共用一条执行线。任务进入 loop 前会带上测试策略和执行策略：小范围逻辑优先单元测试，触达 CLI/API/Agent 契约或跨模块状态时升级集成/契约验证，触达用户主路径、视觉、小程序、性能、安全或成本风险时升级端到端或专项验证；worker 任务还会显式声明 `write-scope`、`local-verify` 和 `integration-owner`，方便主 Agent 在不同窗口里做并行分配。固定 70/20/10 只作健康形状参考，不作硬门禁。真实 `openprd loop --run --agent codex` 会在启动子会话前运行 `codex --version`；如果 Codex 平台原生可选依赖缺失，OpenPrd 会停止 `codex exec` 并输出缺失包、诊断和 `npm install -g @openai/codex@latest`，只有用户显式使用 `doctor --tools codex --fix` 或 `loop --run --agent codex --repair-agent` 时才执行全局修复。新增或修改代码文件前后，Agent 先对 touched files 运行 `openprd dev-check` 或 `scripts/openprd-dev-check.mjs`；当出现需要关注的文件时，最终回复必须以 **后续建议** 为标题，按 🔴 → 🟠 → 🟡 排序说明影响位置、关注程度、本次处理和后续建议，让用户一眼看到历史大文件与本轮处理边界；大界面改动必须先有 3 方向效果图评审和用户选定方向；如果界面任务已有参考效果图、设计稿或用户给图，阶段性完成后必须截实现图并运行 `openprd visual-compare --reference/--actual --locale <zh-CN|en>` 输出左右对比 JPG，查看后继续复刻到无明显差异；如果参考图是一张整板、网格图或多对象组合，还必须先运行 `openprd visual-prepare` 生成 reference-set、contact sheet 和 compare-plan，再逐项比对；如果没有参考图但改动界面，动手前必须截修改前图，完成后用同一入口、视口、账号和数据状态截修改后图，并运行 `openprd visual-compare --before/--after --locale <zh-CN|en>` 输出修改前后自检 JPG，查看后确认预期变化和未改区域漂移；完成前还要检查 `docs/basic/`、文件说明书、文件夹说明书、结构化日志、业务护栏、测试策略、执行模式、视觉方案确认、视觉复刻或自检证据和评估证据是否缺失或过期。
- 自我成长：当执行中发现可沉淀项时，不在任务过程中打断 Agent。代码扩展识别这类白名单工具补全会自动补齐并记录；用户偏好、项目协作规矩和 OpenPrd 默认行为先作为候选留存，收工时通过 `openprd grow --review` 集中让用户确认。执行或复盘中被用户采纳的优质外部信源通过 `openprd benchmark observe` 累计到 benchmark candidate，达到阈值后只推荐 approve，不自动晋级为长期来源。
- OpenPrd 自身维护：当需求涉及新增或修改配置类能力时，Agent 先判断是否 grow-aware。代码扩展识别这类白名单工具补全默认直接纳入自动应用；其他高置信可成长能力默认补 candidate type、scope、review/apply 行为；不确定时主动询问用户是否纳入 `openprd grow`。
- 回归与就绪检查：实现完成后运行项目测试、`openprd standards --verify`、`openprd quality --verify`、`openprd run --verify` 和必要的 `doctor`。同时沉淀阶段性回归报告：前端任务优先记录界面验证策略；已有参考图的界面任务还要沉淀 `.openprd/harness/visual-reviews/` 下的 `openprd visual-compare --reference/--actual` 视觉对比图；无参考图但改动界面的任务还要沉淀 `openprd visual-compare --before/--after` 修改前后自检图；后端任务记录脚本或命令验证；`loop --finish` 在最后一个任务完成时会自动执行最终质量门禁，未 production-ready 时不标记最终任务完成。最终输出结构化测试报告、HTML 回归报告和 HTML 质量评估报告。涉及消耗型成本时，还要确认额度绕过、并发请求、越权身份、成本监控、报警阈值和止损动作已覆盖；如果质量报告 `productionReady=false`，最终输出必须先区分 `taskReady` 与 `workspaceReady`，列出缺证据或需关注的本期必测块；若只剩 `feature-coverage`，说明是任务账本或覆盖证据未收口，不能把本次功能表述成失败，也不能宣称整体就绪。
- 复盘学习与经验沉淀：`openprd learn` 先生成证据清单、Agent 上下文和写作提示；标题、大纲、正文、检索练习和工作示例由 Agent 基于证据写入 `learning-content.json` 后再渲染到 `reader.html`。Codex hook 会把“生成学习材料 / 教学材料 / 讲义 / 可长期阅读材料”等学习型交付物按产物形态路由到该流程，并在 Stop 阶段检查本轮是否已有 reader 骨架和 Agent-authored 内容。已验证修复、重复问题、高影响排障或 Agent 误判问题会先通过 `openprd quality --learn --review --from <turn-state|report|diagnostics>` 生成 `.openprd/knowledge/candidates/` 和 `.openprd/knowledge/drafts/` 下的待确认草案，再通过 `openprd quality --learn --from <candidate-dir|report>` promote 为 `.openprd/knowledge/skills/` 下的项目级经验 skill。

## 状态变化

- `run --context` 只产生建议上下文，不自动授权执行；如果显式继续历史会话，恢复顺序是“全局 session registry -> 目标 workspace 的 repo-local 线索 -> change/task/work unit”，而不是“先用当前 workspace 的 active change 猜一个相近目标”。
- `run --context` 和 `run --verify` 默认只面向当前项目主验证状态；`mode=reference` 的 discovery run 不会自动混入主上下文或主校验，参考调研需通过 `openprd discovery` 命令单独查看。
- 默认 hook 保留轻量 PreToolUse 写入门禁，并把 matcher 收窄到直接编辑工具；`Stop` 不做完整工具级遥测，只在本轮结束前回顾 touched files 和 verify/finish 信号。若本轮 touched code files 需要用户关注，Stop 会把带标题的 **后续建议** 表格注入最终回复；否则再判断是否生成项目经验草案。`guarded` 会额外覆盖 shell 工具，只有 `full` profile 才启用更重的全量工具级诊断。hook 只负责注入或打开 requirement-intake 安全通道，不把 matcher 当作最终 L2 判断；Agent 需要先用 `openprd-requirement-intake` 做语义分流，L0 直接处理，L1 给对话内 mini-plan，L2 才进入 PRD/review/change/tasks。带报错、日志、复现、根因排查等故障证据且用户明确要求直接修复的小型 bugfix 不进入需求入口，已打开入口也接受“确认修复”等自然确认词。
- `loop --run`、`tasks --advance`、`discovery --advance`、`finish --commit` 只有在用户明确执行意图下才进入执行状态。
- 深度 discovery 的只读 subagent 输出只作为候选证据，最终状态更新仍由主 Agent 通过 OpenPrd claim、任务和验证流程落地。
- standards 遇到嵌套 Git 参考仓库或 `toolkit-sources` 这类明显外部参考候选时，会先提示候选和归类命令，而不是直接把逐文件说明书缺口作为主阻塞；只有显式归类后才会稳定写入 `externalReferences.paths`。
- `clarify` 会先产出 `intake-reflection.md` 作为可审计的内部自省记录，再用 `inline` 或 `inline-with-checklist` 在 CLI 和对话内输出轻量澄清提纲。澄清阶段不生成 HTML，也不会自动打开浏览器；需要用户正式评审时，统一进入后续 `review.html`。无 active requirement gate 的冷启动工作区，只要还缺关键字段，也会保留这份首轮项目画像记录。
- `synthesize` 会产出稳定版本评审页和 active 快捷入口，交互式 CLI 自动打开稳定版本评审页。评审页用需求概览、需求关系图或需求流程图、四个带重点摘要胶囊的固定评审卡片和底部固定双按钮工具栏，把问题、范围、主流程、业务护栏、风险和开放问题压缩成需求定稿前的人机确认界面；顶部不展示低价值计数和状态噪音。需求关系图使用紧凑高度和小尺寸节点，节点标题应由 Agent 控制在 15 字以内，节点正文左对齐且每个卡片正文应由 Agent 控制在 30 字以内，避免大块居中文案占用视线；HTML 优先读取 `reviewPresentation.mapNodes` 和 `reviewPresentation.flowNodes` 作为图中展示文案，不直接裁剪原文。Agent 可以先运行 `openprd review-presentation . --template` 获取展示文案模板，再用 `openprd review-presentation . --presentation review-presentation.json --fail-on-violation` 校验；通过后用 `openprd capture . --field reviewPresentation --value "$(cat review-presentation.json)" --source agent-inferred` 写入展示层，也可以用随包发布的 `scripts/openprd-review-presentation.mjs` 做同样校验。OpenPrD review 上下文也会输出 `presentationContract`、`presentationFeedback`、`versionId`、`digest` 和 `workUnitId`，让 Agent 重新提炼超限内容并在确认时校验是否仍是本轮需求。“主流程与边界情况”卡片内会用轻量 SVG 小图展示用户旅程、关键步骤、边界情况和恢复路径，再展示明细列表。四个评审卡片的明细分点应由 Agent 写成 `- **摘要内容**：明细一句话`，HTML 会按“加粗摘要 + 一句说明”渲染，缺少该结构时通过 `presentationFeedback` 提醒 Agent 重写。重点摘要胶囊用于扫读，Agent 应控制在 15 个字以内，HTML 侧按内容自适应宽度并完整显示，不使用省略号截断；评审卡片副标题句末不加句号。底部工具栏固定在窗口底部，只保留“需要调整”和“认可方案”两个动作，点击复制内容必须带上版本、摘要指纹、work unit、标题和 OpenPrD review 上下文。面向用户的 review.html 和 diagram HTML 文案不得使用 `freeze` 这类内部词，应改写为“需求定稿前”或“进入实现前确认”。Agent 必须告知稳定 HTML 路径。当前 lane 仍要求人类决策时，确认请求要围绕这份稳定 artifact 发起；`review --mark confirmed` 只用于记录当前精确匹配的 PRD 版本，如果传入的 `--version`、`--digest` 或 `--work-unit` 与快照不一致，确认会被阻断。实现授权和 review 记录是两个独立门禁；就算用户已经说“可以开做”、单纯说“请帮我实现”，或表示“继续实现”，也不能替另一份 artifact 补确认；只有用户明确表示“不需要进行任何确认”的 `silent-record` lane 中，才可以对当前精确匹配的稳定 artifact 直接记录 review，然后继续 change / tasks。
- `change --generate` 只能在最新 PRD review 已确认后执行，并会在生成的任务里写入 `type`、`test-layer`、`test-size`、`test-scope`、`evidence-plan` 以及 `execution-mode`、`parallel-group`、`write-scope`、`owner-role`、`local-verify`、`integration-owner` 元数据。`implementation` 和 `verification` 任务默认使用 `tasks --evidence-required` 作为局部证据门，要求 Agent 传入本任务 evidence 或 waiver，而不是每个任务都运行全局 `run --verify`；旧版任务若仍写着 `verify: openprd run . --verify`，在 `tasks --verify` 上下文中也会转为局部 evidence 门，不触发 workspace quality churn；`documentation` 和 `governance` 任务用于保证文档和流程收口不会被误计为实现复杂度；测试策略元数据用于 evidence routing，不用于机械比例考核；执行策略元数据用于主 Agent 做 worker 分片和最终审查，不直接触发自动并行。
- `loop --finish` 会同时产出 Markdown 测试报告和 HTML 回归报告，并在报告里记录本任务测试策略、执行命令、结果和证据路径，用于复盘、交接和重复验证。
- 大界面改动进入实现前，`.openprd/harness/visual-reviews/` 应先保存 3 方向横向效果图评审大图和用户选定方向记录；该评审位于需求分流之后、PRD 定稿或实现开工之前。
- `visual-prepare` 会把确认后的整板、网格图或多对象参考图整理成 reference-set，生成 crops、contact sheet、compare-plan 和 board 模板，作为逐项视觉验收的准备层。
- `visual-compare` 会把效果图和实现截图合成左右对比图片，也会把修改前和修改后截图合成自检图片，默认输出 JPG 到 `.openprd/harness/visual-reviews/`；它只用于实现阶段视觉证据，不能替代 Codex Image 2 的图片或效果图生成，也不能替代大界面改动的实现前方向评审。同构列表、卡片、网格或表格的 `alignment-board` 必须同时覆盖容器轨道和内部内容槽位；只量外框、列宽或行顶不算完整对齐验收。单个素材、图标、头像、徽标、按钮图形或图片内部居中/视觉重心判断使用 `centering-board`，必须同时记录画布中心、主体外接框中心和视觉重心偏移；单张截图或主观判断不算完整居中验收。
- `dev-check` 只检查 Agent 本次传入的代码文件，不扫描全仓库；它会把需要关注的文件转成产品和业务可读的关注程度：🟡 低风险｜建议留意、🟠 中风险｜建议优先关注、🔴 高风险｜需要先处理。无需额外说明的改动不进入用户可见提醒。
- `dev-check` 的 JSON 结果包含 `wrapUp.title`、`wrapUp.rows`、`wrapUp.markdownTable` 和 `wrapUp.markdownBlock`；当 touched files 需要用户关注时，CLI 会打印“后续建议”，Stop hook 会把同一张面向产品和业务的影响提示表交给最终回复使用，并按 🔴 → 🟠 → 🟡 的关注程度排序。
- `dev-check` 若遇到未配置扩展名但内容明显像代码，会先按代码候选参与本次分级。置信度达到自动补全阈值时直接把扩展名追加到 `.openprd/standards/config.json` 并在 `.openprd/growth/accepted.json` 留证据；置信度不足或不属于工具识别补全时，只写入 `.openprd/growth/candidates.jsonl` 等收工复盘。
- 新增配置能力时，状态变化不止是“写入代码”：Agent 还要给出该能力是否纳入 growth 的判断。代码扩展识别这类白名单工具补全可直接自动固化；用户偏好、项目协作规矩和 OpenPrd 默认行为等需要用户决定的内容停在收工确认状态。
- `benchmark observe` 更新 `.openprd/benchmarks/inbox/` candidate 的累计采纳次数、最近 7 天滚动采纳统计、最近使用时间和证据；`benchmark approve` 是唯一把来源移入 approved registry 的动作。`benchmark verify` 会独立校验推荐阈值、approve command 和“只推荐、不自动晋级”的控制状态是否漂移。
- `quality --verify` 会产出 `.openprd/quality/reports/<id>.json` 和 `.openprd/quality/reports/<id>.html`，HTML 回归测试报告是阶段性质量查看的主要产物，包含整体结果、需求模块结果、测试块状态、分层测试策略矩阵、本次执行证据和业务成本与滥用护栏。
- `quality --learn --review` 会先生成 knowledge candidate 和 draft skill；`quality --learn` 再把已确认的模式抽象为 incident、pattern 和项目经验 Skill，形成“问题修复 -> 验证 -> 草案 -> 沉淀 -> 下次触发”的项目级认知循环。
- `learn` 默认进入 `awaiting-agent-content` 状态，只提供 HTML 阅读器骨架和 Agent 写作工具包；传入 `--content-json` 后才进入 `agent-authored` 状态并展示正文。
- 新增或修改代码文件会先触发研发期行数回顾；新增或修改任何文件还会触发文档影响判定：缺失文档进入补齐状态，已有但受影响的文档进入更新状态，无需更新时要说明理由。
- standards、validate 或 doctor 失败时，状态回到修复阶段，不应声明实现完成。

## 维护规则

- 每次用户流程、页面跳转、任务状态或异常处理发生变化后，必须检查并更新本文件。
